React Hooks Explained

useState, useEffect & useRef — from zero to confident, with real examples and analogies.

🪝 What Are React Hooks?

Imagine you are building a house. The house is your React component. But a bare house needs electricity, plumbing, and memory to actually be useful. React Hooks are those built-in systems you can "plug into" to give your component superpowers.

In simple words: Hooks are special functions that let your components remember things, react to changes, and directly interact with the page.

🏡 Real-life Analogy

A component without hooks is like a sticky note — it shows information but can't do anything on its own. Hooks are like turning that sticky note into a smartphone — now it can remember your inputs, connect to the internet, and interact with the world!

The 3 Rules of Hooks

  • 1

    Only call hooks at the top level of your component. Never inside loops, conditions, or nested functions.

  • 2

    Only call hooks inside React function components (or your own custom hooks). Not in regular JS functions.

  • 3

    Hook names always start with "use" — that's the convention React uses to recognize them.

useState

Gives your component a memory — it remembers data between renders.

What is it? 🧠

When your component needs to remember a value — like the number on a counter, the text in an input, or whether a checkbox is checked — you use useState. Whenever the state changes, React automatically re-draws (re-renders) the component to show the new value.

🛒 Real-life Analogy

Think of useState like a whiteboard in your kitchen. You write the number of eggs on it. Whenever you use one egg, you erase the old number and write the new one. Every time the whiteboard changes, everyone in the kitchen instantly sees the update.

Syntax

SYNTAX
import { useState } from 'react';

// Pattern:
const [value, setValue] = useState(initialValue);

// value     → the current state (what you READ)
// setValue  → the function to UPDATE the state
// useState(initialValue) → the starting value
💡 Memory Trick

Think of useState as returning a pair: "the box" and "the way to change the box". You always get two things out.

Example 1 — Counter
JSX
import { useState } from 'react';

function Counter() {
  const [count, setCount] = useState(0); // start at 0

  return (
    <div>
      <p>You clicked {count} times</p>
      <button onClick={() => setCount(count + 1)}>
        Click me!
      </button>
    </div>
  );
}

Step-by-step: How does this work?

  • 1

    useState(0) creates a memory slot called count, starting at 0.

  • 2

    React displays the component: the screen shows "You clicked 0 times".

  • 3

    You click the button. The onClick calls setCount(count + 1).

  • 4

    React sees the state changed. It re-renders the component. Now count is 1.

  • 5

    The screen now shows "You clicked 1 times". This cycle repeats every click.

Example 2 — Toggle (Show/Hide)
JSX
function Spoiler() {
  const [isVisible, setIsVisible] = useState(false);

  return (
    <div>
      <button onClick={() => setIsVisible(!isVisible)}>
        {isVisible ? 'Hide Spoiler' : 'Reveal Spoiler'}
      </button>

      {isVisible &&
        <p>🐉 The dragon was the good guy all along!</p>
      }
    </div>
  );
}

When to use useState ✅

✅ Use it when

You need to track values that change over time AND need to update the UI — counts, form inputs, toggles, selected items, loading states, error messages.

❌ Don't use it when

The value changes but doesn't need to update the screen (use useRef instead), OR the value never changes at all (just use a regular variable).

⚠️ Common Beginner Mistakes

Mistake 1: Modifying state directly — count = count + 1 — This does NOT work. Always use the setter: setCount(count + 1).


Mistake 2: Reading state right after setting it. State updates happen asynchronously. The new value won't be available on the very next line — React batches them for performance.

🧠 Memory Trick
useState = "Remember this for me, React!"

You hand React a value to remember, and ask for it back — plus a way to change it.

useEffect

Runs code after the component renders — like setting up side effects.

What is it? ⚡

useEffect lets you run code after your component appears on the screen. It's perfect for things that happen "on the side" — fetching data from an API, starting a timer, or subscribing to events. These are called side effects because they live outside of rendering.

☕ Real-life Analogy

Imagine you move into a new apartment. Once you're settled in (rendered), you do some setup work: subscribe to Netflix, turn on the Wi-Fi, water the plants. That's useEffect — it's the code you run after you've arrived, not during the move itself.

Syntax

SYNTAX
import { useEffect } from 'react';

useEffect(() => {
  // Your side-effect code goes here

  return () => {
    // Optional: Cleanup code (runs before next effect or unmount)
  };
}, [dependency1, dependency2]); // ← The dependency array

🎯 The Dependency Array — Explained Simply

The second argument of useEffect is the dependency array. It tells React when to re-run the effect. Think of it as a watch list.

3 CASES
// Case 1: No array — runs after EVERY render
useEffect(() => { console.log('runs always!'); });

// Case 2: Empty array [] — runs ONCE (on mount only)
useEffect(() => { console.log('runs once on load'); }, []);

// Case 3: [value] — runs when 'value' changes
useEffect(() => { console.log('count changed!'); }, [count]);
💡 Dependency Array Trick

Ask yourself: "What values does my effect use?" Those values go in the array. If you use userId inside the effect, put [userId] in the array. If you use nothing, use [].

Example 1 — Page Title Updater
JSX
function PageTitleCounter() {
  const [count, setCount] = useState(0);

  useEffect(() => {
    // This runs every time 'count' changes
    document.title = `You clicked ${count} times`;
  }, [count]); // ← Watches 'count'

  return (
    <button onClick={() => setCount(count + 1)}>
      Click ({count})
    </button>
  );
}

Step-by-step: How does this work?

  • 1

    Component renders for the first time. count is 0.

  • 2

    After rendering, useEffect runs — sets document title to "You clicked 0 times".

  • 3

    You click the button. count becomes 1. Component re-renders.

  • 4

    React checks the dependency array [count] — count changed? Yes! So it runs the effect again.

  • 5

    Title is now "You clicked 1 times". Cycle repeats on every click.

Example 2 — Fetch Data on Load
JSX
function UserProfile({ userId }) {
  const [user, setUser] = useState(null);

  useEffect(() => {
    // Fetch user data when userId changes
    fetch(`/api/users/${userId}`)
      .then(res => res.json())
      .then(data => setUser(data));
  }, [userId]); // ← Re-fetch if userId changes

  if (!user) return <p>Loading...</p>;
  return <p>Hello, {user.name}!</p>;
}

The Cleanup Function 🧹

Sometimes your effect creates something that needs to be cleaned up — like a timer or an event listener. You return a function from useEffect to handle this:

CLEANUP
useEffect(() => {
  const timer = setInterval(() => {
    console.log('tick!');
  }, 1000);

  return () => {
    clearInterval(timer); // 🧹 Stop timer when component leaves
  };
}, []);

When to use useEffect ✅

✅ Use it when

You need to fetch data, start timers, add event listeners, update the document title, or do anything that reaches outside React's world.

❌ Don't use it when

You're just computing a value from existing state/props — do that directly during render. Don't put calculations inside useEffect just to store them in state (causes double renders).

⚠️ Common Beginner Mistakes

Mistake 1: Missing the dependency array — your effect runs after every single render, which can cause infinite loops (especially if you call setState inside).


Mistake 2: Using a stale value inside the effect because you forgot to add it to the dependency array. If your effect uses count, always add count to the array.

🧠 Memory Trick
useEffect = "After you show up, do this."

It's the hook that runs after the component renders. Side effects happen on the side — after the main show.

useRef

A silent note — stores a value or grabs a DOM element without triggering re-renders.

What is it? 📌

useRef is like a sticky note attached to the back of your component — you can write on it and read it, but changing it does NOT cause the component to re-draw. It has two main uses: 1) remembering a value silently, and 2) directly accessing a DOM element (like an input box).

🗒️ Real-life Analogy

Imagine a sticky note on the back of your laptop screen. You can jot things on it — a phone number, a reminder — and check it anytime. But nobody else sees it change, and nobody reacts to it. That's useRef: private, silent, always there.

Syntax

SYNTAX
import { useRef } from 'react';

const myRef = useRef(initialValue);

// The ref object always has one property: .current
myRef.current // ← read the value
myRef.current = 'new value'; // ← change the value (no re-render!)

// For DOM access:
<input ref={myRef} /> // ← attach ref to an element
Example 1 — Auto-Focus an Input
JSX
function SearchBar() {
  const inputRef = useRef(null); // point at nothing yet

  useEffect(() => {
    inputRef.current.focus(); // focus the real DOM input
  }, []); // only once, on load

  return (
    <input
      ref={inputRef}
      placeholder="Type to search..."
    />
  );
}

Step-by-step: How does this work?

  • 1

    useRef(null) creates a ref object: {"{ current: null }"}.

  • 2

    The ref={"{inputRef}"} on the input tells React: "When this input is created in the DOM, attach it to inputRef.current".

  • 3

    After the component renders, useEffect runs. Now inputRef.current points to the real HTML input element.

  • 4

    inputRef.current.focus() calls the browser's native focus — the cursor jumps into the input automatically!

Example 2 — Count Renders Without Re-rendering
JSX
function RenderCounter() {
  const [text, setText] = useState('');
  const renderCount = useRef(0);

  useEffect(() => {
    renderCount.current += 1; // silently increment, no re-render!
  });

  return (
    <div>
      <input value={text} onChange={e => setText(e.target.value)} />
      <p>This component rendered {renderCount.current} times.</p>
    </div>
  );
}

When to use useRef ✅

✅ Use it when

You need to access a DOM element directly (focus, scroll, measure), OR store a value that changes but must not trigger re-renders (previous value, timer ID, render count).

❌ Don't use it when

Your value change needs to update the screen. If you change ref.current, the UI won't update. That's what useState is for.

⚠️ Common Beginner Mistakes

Mistake 1: Using useRef when you actually need useState. If you write myRef.current = 5 and expect the screen to show 5 — it won't.


Mistake 2: Accessing ref.current during the render phase (before the component has mounted). Always access refs inside useEffect or event handlers, not at the top level of the component.

🧠 Memory Trick
useRef = "A secret pocket — no one reacts to it."

It stores things quietly. It doesn't shout. React ignores it during rendering.

⚔️ Bonus: useState vs useRef

This confuses so many beginners! Here's the clearest way to think about it:

Feature useState useRef
Causes re-render? ✅ Yes — always ❌ No — never
Updates the UI? ✅ Yes ❌ No
Can access DOM? ❌ No ✅ Yes (via ref={"{ref}"})
Persists across renders? ✅ Yes ✅ Yes
Use for Counter, input text, toggle Focus, timers, scroll, render count
Golden rule If it shows on screen → useState If it's invisible/silent → useRef
💡 One-line rule

useState = "The user needs to SEE this change."  |  useRef = "I need to remember this, but don't bother redrawing."

🚀 Mini Project: Smart Search Input

A real-world component that uses all three hooks together — with auto-focus, a live character count, and a search history tracker.

useState useEffect useRef

Here's what this component does: it auto-focuses the search box, tracks what you type, updates the page title live, and remembers the previous search — all three hooks working together.

FULL COMPONENT
import { useState, useEffect, useRef } from 'react';

function SmartSearch() {
  // ① useState: remember the current search text
  const [query, setQuery] = useState('');

  // ① useState: remember the list of past searches
  const [history, setHistory] = useState([]);

  // ② useRef: grab the real input DOM element
  const inputRef = useRef(null);

  // ② useRef: silently remember the previous query
  const prevQuery = useRef('');

  // ③ useEffect: auto-focus the input on load
  useEffect(() => {
    inputRef.current.focus();
  }, []); // empty array = runs once

  // ③ useEffect: update the page title whenever query changes
  useEffect(() => {
    document.title = query
      ? `Searching: ${query}`
      : 'Smart Search';
  }, [query]); // runs when query changes

  const handleSearch = () => {
    if (!query.trim()) return;

    // Save previous query via ref (no re-render)
    prevQuery.current = query;

    // Add to history via state (triggers re-render to show it)
    setHistory(prev => [query, ...prev.slice(0, 4)]);
    setQuery('');
  };

  return (
    <div style={{ padding: '20px' }}>
      <h2>🔍 Smart Search</h2>

      <input
        ref={inputRef}         {/* useRef attaches here */}
        value={query}          {/* useState controls this */}
        onChange={e => setQuery(e.target.value)}
        placeholder="Search..."
      />

      <button onClick={handleSearch}>Search</button>

      <p>Characters: {query.length}</p>

      {prevQuery.current &&
        <p>Last searched: {prevQuery.current}</p>
      }

      <h3>Recent Searches:</h3>
      <ul>
        {history.map((item, i) =>
          <li key={i}>{item}</li>
        )}
      </ul>
    </div>
  );
}

How all three work together:

  • S

    useState manages query (current text) and history (past searches) — both need to update the screen.

  • E

    useEffect auto-focuses the input on load ([]), and updates the browser tab title each time query changes ([query]).

  • R

    useRef grabs the DOM input for focus control, and silently tracks prevQuery without triggering any re-renders.

💡 Why not use useState for prevQuery?

Because we don't need the UI to immediately react to prevQuery changing. It's internal bookkeeping. Putting it in useRef is more efficient — no extra renders.

🎯 Quick Decision Guide

Next time you're not sure which hook to reach for, ask yourself these questions:

  • ?

    "Does this value need to update the screen when it changes?" → Yes: useState | No: useRef

  • ?

    "Do I need to fetch data, start a timer, or talk to the outside world?" → Yes: useEffect

  • ?

    "Do I need to directly access an HTML element (like an input or div)?" → Yes: useRef

  • ?

    "Do I need to remember something only once, when the component first loads?" → Yes: useEffect with []
